
# Chat Application System Documentation

## 1.1 Project Overview
This Chat Application is a client-server real-time messaging system developed to facilitate communication between multiple users in a local network environment. It provides support for both public group chats and private one-on-one conversations, along with user authentication and message persistence using a local SQLite database to avoid the need for complex administrative overhead or external server installations on individual workstations.

## 1.2 System Design

### 1.2.1 System Architecture
The Chat Application is built on a client-server architecture using the .NET 8 framework. The system uses TCP/IP for reliable, ordered message delivery on port 8888, eliminating the need for UDP-based server discovery in this version. The client application is developed using Windows Forms (WinForms) for the user interface, while the server is a console application that handles client connections, authentication, and message routing.

### 1.2.2 Data Storage Design
The database is managed using Microsoft.Data.Sqlite 10.0.10 library, which provides high-performance reads and writes essential for real-time messaging while keeping the application's footprint minimal. The database file is named chat.db and is automatically initialized in the application's root directory when the server first runs. The storage design is centered around two primary relational tables: Users and Messages. The Users table manages identity and security by storing unique user identifiers as the primary key and BCrypt hashed passwords. The Messages table archives every interaction, including a unique message ID with autoincrement, sender username, receiver username (nullable for public messages), message content, ISO 8601 formatted timestamp, and an integer flag indicating whether the message is private (1) or public (0).

### 1.2.3 Core Components
The Chat Application solution is architected with a modular project structure designed to ensure the codebase remains maintainable, scalable, and organized through a clear separation of concerns. The system is divided into three distinct components, each serving a specialized role in the communication ecosystem: ChatApp.Shared, ChatApp.Server, and ChatApp.Client.
ChatApp.Shared acts as the foundational library for the entire application. It defines the custom communication protocol layer and the network packet structures that both the client and server nodes must use identically to exchange data. By housing shared models, constants, and JSON serialization handlers, this component ensures that information is encoded and decoded consistently across the network, preventing synchronization errors.
ChatApp.Server is the standalone core engine of the system. It is responsible for the critical logic of network management, including handling persistent TCP connections on port 8888 and maintaining the active user state. Beyond basic network routing—where it determines whether to broadcast public messages or forward private ones to specific recipients—the server component also manages mandatory user authentication and interacts directly with the SQLite database to persist message history.
ChatApp.Client represents the graphical presentation layer developed using Windows Forms (WinForms). This component contains all user interface elements and client-side network logic. It functions by mapping UI events, such as clicking a send button, directly to background service request protocols to ensure the UI thread remains responsive while the background threads handle the complexities of message sending and receiving.

## 1.3 Development Process
The development of the Chat Application was a structured process focused on balancing high-level technical performance with end-user accessibility. The project utilizes a modern technology stack, with C# 12 as the primary programming language. This version of C# was specifically selected for its advanced asynchronous programming features, such as async/await, which are critical for maintaining a responsive system during high-latency network operations. The application is built on the .NET 8 framework, providing a robust, long-term support runtime and extensive class libraries that facilitate rapid development and broad compatibility. For the user interface, Windows Forms (WinForms) was chosen for its seamless integration with the Windows operating system and its ability to provide access to native controls, ensuring a familiar experience for users.
The development process emphasized a modular architecture to keep the codebase maintainable and scalable. The solution is partitioned into three distinct projects: ChatApp.Shared, ChatApp.Server, and ChatApp.Client. The Shared library defines the custom communication protocol layer, network packet structures, and JSON serialization handlers used identically by both nodes to ensure data consistency. The Server project houses the core engine responsible for TCP connection handling, maintaining the active user list, and managing user authentication. The Client project maps UI events directly to background service request protocols, ensuring a clean separation between the presentation and networking layers.
A central pillar of the development process was the application’s design philosophy, which is rooted in minimalism and functionality. Developers adhered to core design principles to create an intuitive, distraction-free atmosphere using the Segoe UI font and a professional blue-based color scheme. Feedback mechanisms, such as hover states, were implemented to provide immediate visual cues to the user regarding the results of their actions.

## 1.4 Testing and Evaluation
Testing for the Chat Application is categorized into four primary domains to ensure system integrity: unit testing, integration testing, functional testing, and user acceptance testing.

### 1.4.1 Unit Testing
Unit testing for the Chat Application focuses on the isolated verification of critical logic components housed within the ChatApp.Shared library. This includes testing the JSON serialization and deserialization handlers to confirm that complex data objects like the Message model are accurately converted into length-prefixed byte streams for network transmission. Additionally, tests can verify the database context operations to ensure proper interaction with the SQLite database. By rigorously testing these standalone modules, the development team ensures that the core data processing and protocol logic remain stable and reliable throughout the application's lifecycle.

### 1.4.2 Integration Testing
Integration testing for the Chat Application focuses on the seamless interaction between the Client, Server, and Shared projects. This phase specifically validates the connection and authentication handshake, ensuring the client can successfully connect to the server via the specified IP address and port 8888. Tests verify the establishment of a persistent TCP connection and the correct execution of the login and registration processes, confirming that credentials are properly authenticated against the SQLite database and that active sessions are registered correctly.

### 1.4.3 Functional Testing
Functional testing for the Chat Application verifies that the application's core features operate according to defined business rules and user requirements. This includes testing the registration flow, ensuring the system properly stores new user credentials and rejects duplicate usernames. The authentication process is validated by confirming that valid credentials grant access to the main chat window, while invalid login attempts trigger appropriate error messages to prevent unauthorized access. Additionally, testing covers messaging functionality, ensuring public messages broadcast to all users and private messages reach specific recipients, and that all messages are correctly stored in the database. Finally, the graceful disconnection process is verified to ensure users can exit the application properly.

### 1.4.4 User Acceptance Testing
User Acceptance Testing (UAT) evaluates the Chat Application against its core design principles of minimalism and functionality. Testers confirm that the minimalist interface provides an intuitive experience where all interactive elements are clearly labeled. UAT verifies the consistency of the blue-based color scheme and the accessibility of the Segoe UI typography. Crucially, it assesses if immediate feedback, such as hover animations, effectively informs users of their actions, ensuring a distraction-free and professional communication environment.

## 1.5 Project Results
The successful completion of the Chat Application project has resulted in a fully functional, synchronized client-server communication ecosystem that meets all architectural and design specifications outlined in the project requirements. The primary result is a robust messaging platform capable of facilitating both public group discussions and private one-on-one interactions with high reliability and minimal latency.
One of the standout technical results is the reliable connectivity established through the TCP/IP protocol on port 8888. The implementation of asynchronous programming has proven successful in maintaining transaction integrity, with the server effectively managing concurrency by processing logic asynchronously to prevent hanging connections and maintain state synchronization across all nodes.
The user interface results demonstrate a high level of usability and aesthetic consistency. By utilizing a blue-based color scheme and Segoe UI font, the application achieves a modern, professional look that remains distraction-free. The split-view layout—separating contact management in the left sidebar from the active conversation in the right panel—optimizes screen space and user workflow. Additionally, the system effectively maintains the clarity of the primary chat feed by focusing solely on message content.
From a performance and security perspective, the project delivered a portable SQLite 3 persistence layer that initializes automatically upon first execution. This result ensures that the system is ready-to-run in classroom environments without external database dependencies. Security testing confirmed that user credentials are encrypted using BCrypt before being committed to the chat.db file, providing a solid foundation for data protection.

## 1.6 Data Storage Testing
Data storage testing for the Chat Application is essential to verify the integrity, persistence, and security of the system's underlying SQLite 3 database. The primary objective of these tests is to confirm that the chat.db file reliably serves as a ready-to-run record-keeping mechanism for all user identities and communication history.

### 1.6.1 Central Storage Testing
Central Storage Testing targets the server-side data layer. Tests verify that when the server process initiates, it correctly locates or constructs the necessary database schema. Critical security testing confirms that user passwords undergo BCrypt hashing before storage, ensuring that sensitive credentials are never kept in plain text. Furthermore, the system is tested to ensure that every message processed by the server is successfully archived in the Messages table with a unique ID and a standardized ISO 8601 UTC timestamp, facilitating accurate message delivery and audit trails.

### 1.6.2 Local Storage Testing
Local Storage Testing focuses on the client’s ability to retrieve and display message history. Testing confirms that when a user logs in, the server correctly loads and sends the public chat history to the client for display. Additionally, these tests verify that the client correctly handles incoming messages and displays them in the appropriate chat view, whether public or private. This comprehensive testing ensures the data layer remains performant, supporting the rapid reads and writes required for real-time messaging.

## 1.7 System Performance Evaluation
The performance of the Chat Application is primarily evaluated based on its ability to maintain client-side responsiveness and ensure the user interface never freezes during network operations. This is achieved through background multi-threading, which offloads the network receive loop and packet processing to secondary threads, allowing the UI thread to remain fluid even during periods of high latency. Additionally, the system leverages the Microsoft.Data.Sqlite library, which facilitates the rapid reads and writes essential for real-time messaging while keeping the server’s footprint minimal. Asynchronous programming further optimizes transaction integrity by preventing blocking calls and allowing efficient handling of multiple connections simultaneously. Finally, the use of .NET 8 and C# 12 provides a high-performance runtime for efficient asynchronous communication.

## 1.8 Usability Evaluation
The Chat Application’s usability is anchored in a philosophy of minimalism and functionality, emphasizing clarity, consistency, and accessibility. The interface features a modern aesthetic using the Segoe UI font and a professional blue-based color scheme. Its split-view layout optimizes workflow by separating the online users list in the left sidebar from the active conversation in the right content panel. Interactive elements like multi-line input and immediate visual cues, such as hover animations, ensure an intuitive and distraction-free user experience.

The development and testing of the C# Windows Forms Desktop Chat Application have successfully achieved the main objectives of the project. In conclusion, the Chat Application meets the required functional and technical specifications and provides a strong foundation for future improvements and additional features.

## 1.11 Limitations and Future Improvement
While the Chat Application provides a robust framework for local communication, several technical and functional limitations offer clear paths for future enhancement. A significant technical constraint is the disconnection process, which currently operates on a best-effort basis. If a user’s application terminates abruptly or network conditions are poor, the logout signal may never reach the server, potentially leaving ghost sessions in the active user list until they time out.
From a functional perspective, the current system does not include UDP-based server discovery, requiring users to manually enter the server IP address. There is also no presence notification system to inform users when others join or leave the chat, no support for user roles or display names beyond the basic username, no announcement feature for broadcasting important messages, and no user profile editing capabilities. Addressing these limitations will transition the Chat Application into a more comprehensive collaborative platform.
